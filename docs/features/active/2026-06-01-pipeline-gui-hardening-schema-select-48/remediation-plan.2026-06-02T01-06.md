## Design Direction (planner-owned, follows the F1 required-fix shape)

The defect: `ThreadedRunner` stores the active thread/worker/receiver in single overwriteable attributes
(`_thread`/`_worker`/`_receiver`), never wires `thread.finished -> worker.deleteLater`, never calls
`thread.wait()`, and has no application-shutdown cleanup. The worker `QObject` (affinity = worker thread)
is therefore destroyed by Python GC on the GUI thread, raising
`QBasicTimer::stop: Failed. Possibly trying to stop from a different thread`.

The fix preserves the existing queued-connection `_RunnerReceiver` pattern (AC-6) unchanged — no direct
cross-thread outcome delivery is reintroduced — and adds correct lifecycle management:

1. `thread.finished.connect(worker.deleteLater)` so the worker is destroyed on its own thread.
2. Replace the single-attribute storage with a collection (a `set` or `dict` of active dispatch records,
   each bundling thread + worker + receiver). On `thread.finished`, wire `thread.deleteLater` and remove
   that record's entry from the collection so a second dispatch cannot drop a still-running prior thread
   and so the records are released only after the thread has finished.
3. Add an `await_active(timeout_ms)` (or equivalently named) public method on `ThreadedRunner` that quits
   and `wait()`s every active worker thread, for use at application shutdown.
4. Wire the shutdown hook in a new sibling module `src/gui/_shutdown_wiring.py` (following the
   `_run_wiring.py` / `_schema_list_wiring.py` precedent) that connects `QApplication.aboutToQuit` to a
   closure calling the runner's `await_active`. `app.py` is AT the 500-line cap (500), so all shutdown
   logic lives in the sibling module; `app.py` gains only one import and one call site.
