## Reused Modules (do not duplicate, do not modify)

- `src/mix_pipeline.py` — CLI `main`, `build_parser`, `_import_sources`, `_persist_all`, `_print_summary` are unchanged. `PipelineService` calls the loaders and `run_transforms` directly.
- `src/mix_pipeline_run.py` — `run_transforms(aop_vs_le, sku_lu, mix_base, le_raw)` called directly by `PipelineService.run_pipeline`.
- `src/mix_lookups.py` — `build_customer_lu`, `build_aop_norm`, `build_le_norm`, `build_aop_vs_le`, `build_mix_base`.
- `src/mix_transforms.py` — `pivot_le`, `pivot_aop`.
- `src/normalize_le.py` — `load_source(path, sheet_name)` -> `normalize(df)` -> `validate_tieouts(source, output)`.
- `src/load_aop.py` — `load_aop(source, *, sheet="AOP1")`.
- `src/load_skulu.py` — `load_skulu(source, *, sheet="SKU_LU")`.
- `src/pandas_io.py` — `read_excel_sheet`, `read_table`, `write_table` typed I/O boundary; exporter and db-service implementations follow this typed-Protocol-view (`cast`) pattern.
- `tests/le_fixtures.py`, `tests/aop_fixtures.py` — these modules supply the LE `BytesIO` workbook builder (`build_workbook`) and the AOP `BytesIO` workbook builder (`build_aop_workbook`), plus the `PersistentConnection`/`patch_connect` `sqlite3.connect(":memory:")` round-trip pattern, all reused by GUI service/integration tests. They do NOT provide any SKU_LU workbook builder and do NOT provide a combined LE+AOP+SKU_LU single-workbook fixture; no shared SKU_LU builder exists in the test suite (the only SKU_LU `BytesIO` builders are private helpers inside `tests/test_load_skulu.py` and `tests/test_mix_pipeline.py`, which the GUI tests do not import). Any GUI test needing a SKU_LU workbook or a combined LE+AOP+SKU_LU single-workbook fixture MUST fabricate it in-file within the GUI test module using the same openpyxl/`BytesIO` pattern: header on Excel row 1 with columns `SKU`, `SKU Description`, `Category`, `International`, and fabricated values only (no real `SKU Description`/`Category` values). Fabricating these in-file does not add a shared fixture file and does not change the per-phase test-file budget.

## Confidentiality Invariant (per-phase acceptance check)

No real `SKU Description` or `Category` values, customer names, SKU numbers, prices,
or discounts may appear in any new source file, test file, fixture, or doc.
Fabricated values only (for example `Acme Foods`, `SKU-1001`, `Example-A`,
`Category X`). The `Country` values `US` and `Canada` are not secret.

## Dependency and Tier Invariants

- `pytest-qt` is the single approved new dependency (dev group), added in Phase 1.
- Every new `src/gui/**` module MUST be classified in `quality-tiers.yml` before CI
  passes; an unclassified project fails the `tier-classification` stage. Tier additions
  are made in the phase that first creates each module so the repository never holds an
  unclassified module across a completed phase.
- Headless Qt tests run under `QT_QPA_PLATFORM=offscreen`, set in `tests/gui/conftest.py`.

## Toolchain Loop (run per implementation phase, in order)

`poetry run black .` -> `poetry run ruff check .` -> `poetry run pyright` ->
`poetry run pytest --cov --cov-branch --cov-report=term-missing`. Restart from Black
if any step fails or changes files. Qt tests require `QT_QPA_PLATFORM=offscreen`
(provided by the `tests/gui/conftest.py` fixture).
